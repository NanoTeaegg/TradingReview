# 持仓股管理与复盘工具 — 技术方案

> 文档状态:已定稿(技术路线)
> 最后更新:2026-05-29
> 说明:功能 1-3 + 数据源 + 交易导入为本期落地范围;功能 4(大模型复盘)、功能 5(新闻轮询推送)在本文档中**仅预留数据表与接口**,具体实现交给 Codex。

---

## 1. 项目概述

一个面向 A 股的个人持仓管理与交易复盘工具,核心是把"真实成交数据 + 交易意图 + 大模型复盘"打通。

### 功能范围

| 编号 | 功能 | 本期 | 负责 |
|------|------|------|------|
| 1 | A 股盈亏分析(成交流水 → 持仓 → 盈亏) | ✅ | 本期 |
| 2 | 交易自动录入(解析同花顺导出文件) | ✅ | 本期 |
| 3 | 交易意图收集(标签化记录交易思路) | ✅ | 本期 |
| 4 | 大模型辅助复盘分析 | 🔌 预留 | Codex |
| 5 | 持仓股新闻轮询与推送 | 🔌 预留 | Codex |

## 2. 技术栈

| 层 | 选型 |
|----|------|
| 前端 | Vite + React + TypeScript + Tailwind CSS + shadcn/ui |
| 后端 | Python + FastAPI + SQLAlchemy 2.0(`Mapped[]` 风格) |
| 数据库 | SQLite(开启 WAL),迁移用 Alembic;后期可平滑迁 PostgreSQL |
| 行情数据源 | TuShare Pro 为主 + akshare 兜底 |
| 交易导入 | 本地 `.xls`/`.csv`,GB18030 + Tab 分隔解析 |
| 大模型(功能4) | Deepseek API 或者 交给 Codex |
| 定时任务(功能5) | APScheduler 或者 交给 Codex |

### 选型说明(决策记录)

- **Vite 而非 Next.js**:本工具是登录后才可见的内部应用,无 SEO/SSR 需求;后端是 Python,Next 的全栈能力用不上,纯 SPA 用 Vite 更轻、心智负担更低。
- **shadcn/ui 而非 Ant Design**:取其现代审美与可控性。代价是复杂表格(基于 TanStack Table)需自行组装,需接受前期组件成本。
- **SQLite 起步**:写并发弱,**必须开启 WAL 模式**;借助 SQLAlchemy,后期迁 PostgreSQL 接近零成本。

## 4. 数据源方案

### TuShare Pro(主)
- **注意:积分制**。免费初始积分低,部分接口有积分门槛(120/2000/5000 不等),且有每分钟调用频率限制。
- 日线行情、股票基础信息一般够用;分钟级与高级接口可能受限。
- 用途:个股最新价/收盘价(算市值与浮动盈亏)、股票名称/基础信息。

### akshare(兜底)
- 完全免费、无积分门槛;接口稳定性/规范性略弱。
- 用途:TuShare 受限时的行情兜底;**个股新闻**(供功能 5)。

> 策略:封装统一的 `MarketDataProvider` 接口,内部 TuShare 优先、失败 fallback 到 akshare,上层无感知。

## 5. 交易导入规格(基于真实样本)

样本文件:`20260421_20260528_Atrading.xls`(同花顺导出)。

### 文件格式(实测)
- **编码**:GB18030
- **结构**:纯文本,**Tab(`\t`)分隔**,**CRLF(`\r\n`)换行**
- **非** HTML、**非** 二进制 Excel
- 表头:`成交日期	证券代码	证券名称	买卖标志	成交价格	成交数量	成交金额	交易市场	摘要`

### 样本数据
```
成交日期    证券代码  证券名称   买卖标志  成交价格   成交数量  成交金额     交易市场  摘要
20260528   001309   德明利     证券买入  661.545   400      264618.000  深Ａ
20260528   300604   长川科技   证券买入  219.010   1000     219010.000  深Ａ
20260528   601138   工业富联   证券买入  72.394    3200     231662.000  沪Ａ
```

### 解析规则与坑位
| 字段 | 处理 |
|------|------|
| 成交日期 | `YYYYMMDD` → `date` |
| 证券代码 | **必须按字符串存**(保留前导零 `001309`) |
| 买卖标志 | `证券买入→buy`、`证券卖出→sell`、**`担保品划入→transfer_in`**(信用账户划入,**非买入**,见 §6.2) |
| 成交价格/金额 | **`Decimal`**,不可用 float;以"成交金额"为权威值(价格是被舍入的均价) |
| 交易市场 | 全角归一化 `深Ａ→SZ`/`沪Ａ→SH`;并生成 TuShare `ts_code`:`深Ａ→.SZ`、`沪Ａ→.SH`(如 `601138.SH`、`001309.SZ`) |
| 成交时间/编号 | **文件不提供**,仅日期级;同日同票多笔按文件行顺序保留相对次序,**无法还原真实分时成交顺序** |
| 行尾 | 每行末尾有多余 trailing `\t`,解析时剔除空尾列 |
| 手续费 | **本文件不含**,`fee` 字段预留(默认 0 / null),后续若有交割单含费再补 |

### 健壮性要求
1. 解析器先按 **GB18030 + Tab** 读;若检测到 `<table>`/`<html>` 则 fallback 用 `pandas.read_html()`(兼容其他券商的 HTML 伪装 .xls)。
2. **导入去重**:对每条成交以 `(trade_date, stock_code, side, price, quantity, amount)` 做唯一约束;并对整文件计算 `file_hash` 防止整批重复导入。
3. 解析失败的行单独收集报错返回,不影响其余行入库。

## 6. 数据模型设计

> 两层存储:`raw_import_rows` 保留原始行(追溯/重新解析),`trades` 为标准化的单一事实源。持仓与盈亏由 `trades` **动态计算**,持仓可选做快照缓存。

### 6.1 import_batches(导入批次)
| 字段 | 类型 | 说明 |
|------|------|------|
| id | PK | |
| filename | str | 原文件名 |
| file_hash | str unique | 整文件哈希,防重复导入 |
| period_start / period_end | date | 文件覆盖区间(可从文件名解析) |
| row_count | int | 成功导入行数 |
| imported_at | datetime | |

### 6.2 raw_import_rows + trades(原始行 + 标准化流水)

**raw_import_rows(原始导入行)** — 逐行原样保存,便于追溯与重新解析
| 字段 | 类型 | 说明 |
|------|------|------|
| id | PK | |
| import_batch_id | FK | |
| row_no | int | 文件内行号(保留文件原始顺序) |
| raw_text | str | 该行原始文本 |
| parsed | bool | 是否成功标准化为 trade |
| error | str null | 解析失败原因 |

**trades(标准化交易流水)** — 单一事实源
| 字段 | 类型 | 说明 |
|------|------|------|
| id | PK | |
| trade_date | date | 成交日期(仅日期级,文件无成交时间/编号) |
| seq | int | 文件行顺序,用于同日多笔的相对排序 |
| ts_code | str | TuShare 格式,如 `601138.SH` / `001309.SZ` |
| stock_code | str(10) | **字符串**,保留前导零 |
| stock_name | str | |
| side | enum(buy/sell/transfer_in) | 见下方映射 |
| price | Numeric(18,4) | 成交价格 |
| quantity | int | 成交数量 |
| amount | Numeric(18,4) | 成交金额(权威) |
| fee | Numeric(18,4) | 手续费,本文件无,预留默认 0 |
| market | enum(SZ/SH) | |
| source | str | 数据来源,如 `ths_xls` |
| remark | str null | 摘要 |
| raw_row_id | FK null | 对应 raw_import_rows |
| import_batch_id | FK | |
| created_at | datetime | |

唯一约束:`(trade_date, stock_code, side, price, quantity, amount)`

买卖标志映射:`证券买入→buy`、`证券卖出→sell`、`担保品划入→transfer_in`

> **担保品划入 ≠ 买入**:信用账户的"担保品划入"是持仓划入/期初持仓事件,映射为 `transfer_in`——**计入持仓数量但不计入买入资金投入**,成本基准按划入价记;绝不可当作 `buy`,否则虚增投入本金、算错盈亏。
> **无成交时间/编号**:文件仅到日期级。同日同票多笔买卖按 `seq`(文件行序)保留相对次序,但**无法还原真实分时成交顺序**,复盘只能做到日级。

## 7. 盈亏计算逻辑

- **持仓成本**:按 **FIFO**(先进先出)逐笔配对计算已实现盈亏;持仓成本采用移动加权平均亦可,二选一并固化。
- **已实现盈亏**:卖出时与对应买入批次配对结算。
- **未实现盈亏(浮动)**:持仓数量 × (最新价 − 持仓均价),最新价取自行情服务。
- **当日盈亏**:持仓数量 × (最新价 − 昨收价)。
- **担保品划入(`transfer_in`)**:增加持仓数量,成本按划入价计入持仓成本,但**不计入买入资金投入统计**,自身不产生已实现盈亏。
- 所有金额运算使用 `Decimal`,展示层再格式化。

## 8. 全局技术约束(铁律)

1. **金额/价格/盈亏一律 `Decimal`(SQLAlchemy `Numeric`),禁止 `float`**——避免账目对不上。
2. **证券代码一律字符串**,禁止整型(前导零)。
3. **SQLite 开启 WAL**:启动时执行 `PRAGMA journal_mode=WAL;`。
4. **Alembic 从第一天接入**,所有表结构变更走迁移。
5. SQLAlchemy 使用 **2.0 风格**(`Mapped[]` / `mapped_column`)。
6. 外部数据源(TuShare/akshare)统一封装、带重试与 fallback,不在业务层直接调用。

## 11. 建议开发里程碑

1. **M1 地基**:后端脚手架(FastAPI + SQLAlchemy + WAL + Alembic)、前端脚手架(Vite + Tailwind + shadcn)。
2. **M2 导入(功能2)**:解析 `20260421_20260528_Atrading.xls` → 入库 `raw_import_rows` + `trades`,含 `transfer_in` 区分、去重与错误行处理。
3. **M3 盈亏(功能1)**:FIFO 计算持仓与盈亏 + 行情服务接入 + 前端持仓/流水页。
4. **M4 意图(功能3)**:意图标签录入与关联成交。
5. **M5 🔌 复盘(功能4)**:交给 Codex,基于 `review_reports` 表与 `input_snapshot` 结构。
6. **M6 🔌 新闻(功能5)**:交给 Codex,APScheduler 轮询 + 推送通道。

## 附:交给 Codex 的对接说明

- **功能 4(复盘)**:后端聚合【某笔/某票的成交数据 + 当时行情 + 关联的 `trade_intents` 标签与思路 + 用户的交易系统规则文本】→ 组装结构化 prompt → 调大模型 API(Deepseek 或其他)→ 落库 `review_reports`(连同 `input_snapshot` 以便复现)。交易系统规则需用户先结构化提供。
