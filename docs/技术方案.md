## 1. 项目概述

一个面向 A 股的个人持仓管理与交易复盘工具,核心是把"真实成交数据 + 交易意图 + 大模型复盘"打通。

### 功能范围

| 编号 | 功能 | 本期 | 负责 |
|------|------|------|------|
| 1 | A 股盈亏分析(成交流水 → 持仓 → 盈亏) | ✅ | 本期 |
| 2 | 交易自动录入(解析同花顺导出文件) | ✅ | 本期 |
| 3 | 交易意图收集(标签化记录交易思路) | ✅ | 本期 |
| 4 | 大模型辅助复盘分析 | 🔌 预留 | Codex |
| 5 | 持仓股新闻轮询与推送 | 🔌 预留 | Codex |

### 前端页面结构（当前版本）

| 路由 | 页面名 | 文件 | 主要内容 |
|------|--------|------|----------|
| `/` | 交易总览 | `Dashboard.tsx` | 绩效摘要四卡片 + 净值图 + 成交流水（含意图抽屉）|
| `/holdings` | 当日持仓 | `TodayHoldings.tsx` | 盘面解析（三栏）+ 持仓明细 + 明日操作思路 |
| `/intents` | 交易复盘 | `Intents.tsx` | 复盘统计看板 + 历史交易列表（Tab 切换）|

## 2. 技术栈

| 层 | 选型 |
|----|------|
| 前端 | Vite + React + TypeScript + Tailwind CSS + shadcn/ui |
| 后端 | Python + FastAPI + SQLAlchemy 2.0(`Mapped[]` 风格) |
| 数据库 | SQLite(开启 WAL),迁移用 Alembic;后期可平滑迁 PostgreSQL |
| 行情数据源 | TuShare Pro 为主 + akshare 兜底 |
| 交易导入 | 本地 `.xls`/`.csv`,GB18030 + Tab 分隔解析 |
| 大模型(功能4) | 双来源可切换:本地 **Ollama** / **OpenAI 兼容云端 API**(Deepseek/通义/Moonshot/OpenAI/智谱等),设置页配置 |
| 定时任务 | FastAPI 进程内轻量后台任务；行情快照空库启动初始化 + 每日 16:10 采集 |

### 选型说明(决策记录)

- **Vite 而非 Next.js**:本工具是登录后才可见的内部应用,无 SEO/SSR 需求;后端是 Python,Next 的全栈能力用不上,纯 SPA 用 Vite 更轻、心智负担更低。
- **shadcn/ui 而非 Ant Design**:取其现代审美与可控性。代价是复杂表格(基于 TanStack Table)需自行组装,需接受前期组件成本。
- **SQLite 起步**:写并发弱,**必须开启 WAL 模式**;借助 SQLAlchemy,后期迁 PostgreSQL 接近零成本。

## 4. 数据源方案

### TuShare Pro(主)
- **注意:积分制**。免费初始积分低,部分接口有积分门槛(120/2000/5000 不等),且有每分钟调用频率限制。
- 日线行情、股票基础信息一般够用;分钟级与高级接口可能受限。
- 用途:个股最新价/收盘价(算市值与浮动盈亏)、股票名称/基础信息。

### akshare(兜底)
- 完全免费、无积分门槛;接口稳定性/规范性略弱。
- 用途:TuShare 受限时的行情兜底;**个股新闻**(供功能 5)。

> 策略:封装统一的 `MarketDataProvider` 接口,内部 TuShare 优先、失败 fallback 到 akshare,上层无感知。

### 盘面情绪快照采集

- 盘面解析数据视为**本地行情快照**,由 `market_sentiment_snapshots` 持久化,不是页面级临时缓存。
- 后端启动后异步检查 `market_sentiment_snapshots`:表内已有任意快照时直接跳过,不调用外部行情;空库时仅初始化当前应展示日期,避免本地开发频繁重启消耗 TuShare/akshare 调用额度。
- FastAPI 运行期间每日 `16:10`(Asia/Shanghai)尝试采集当日快照;若当天是周六/周日则跳过。
- `/api/market/sentiment` 只读取目标日期快照;缺失时执行一次兜底补拉并写库。周六/周日目标日期回退到最近周五。
- 页面重复访问不得直接请求外部行情源;外部 API 只允许通过服务端快照补齐路径触发。

### 持仓价格快照策略

- 持仓数量、成本、浮盈等结果不落持仓快照表,继续由 `trades` 通过 FIFO 动态计算,避免成交或费用规则变化后产生派生数据不一致。
- 持仓最新价/昨收价读取 `market_daily_bars` 日线价格快照,日期与盘面情绪快照日期一致;周六/周日自然读取最近周五价格。
- `/api/positions` 只读 `market_daily_bars`;缺失价格由 `market_sync` 定时/后台按交易日补齐,不在页面请求路径调外部 API。
- `market_daily_bars` 是全市场 A 股日线仓库(TuShare `daily(trade_date=…)` 按日落库);指数仍按代码区间同步;净值/持仓/复盘共享此库。

## 5. 交易导入规格(基于真实样本)

样本文件:`20260421_20260528_Atrading.xls`(同花顺导出)。

### 文件格式(实测)
- **编码**:GB18030
- **结构**:纯文本,**Tab(`\t`)分隔**,**CRLF(`\r\n`)换行**
- **非** HTML、**非** 二进制 Excel
- 已支持导入模板:
  - `ths_standard`: `成交日期	证券代码	证券名称	买卖标志	成交价格	成交数量	成交金额	交易市场	摘要`
  - `ths_contract_export`: `成交日期/交易日期	合同编号	证券代码/股票代码	证券名称/股票名称	操作	成交数量	成交均价	成交金额	交易市场	成交编号	成交时间`

### 样本数据
```
成交日期    证券代码  证券名称   买卖标志  成交价格   成交数量  成交金额     交易市场  摘要
20260528   001309   德明利     证券买入  661.545   400      264618.000  深Ａ
20260528   300604   长川科技   证券买入  219.010   1000     219010.000  深Ａ
20260528   601138   工业富联   证券买入  72.394    3200     231662.000  沪Ａ
```

### 解析规则与坑位
| 字段 | 处理 |
|------|------|
| 成交日期 | `YYYYMMDD` → `date` |
| 证券代码 | **必须按字符串存**(保留前导零 `001309`) |
| 买卖标志 | `证券买入→buy`、`证券卖出→sell`、**`担保品划入→transfer_in`**(信用账户划入,**非买入**,见 §6.2) |
| 成交价格/金额 | **`Decimal`**,不可用 float;以"成交金额"为权威值(价格是被舍入的均价) |
| 交易市场 | 全角归一化 `深Ａ→SZ`/`沪Ａ→SH`;并生成 TuShare `ts_code`:`深Ａ→.SZ`、`沪Ａ→.SH`(如 `601138.SH`、`001309.SZ`) |
| 成交时间/编号 | `ths_contract_export` 提供 `成交时间`(`HH:MM:SS`) 与成交编号;`trades.trade_time` 保存时间,旧模板无时间则为空字符串;同日排序优先 `trade_time`,无时间时回退 `seq/id` |
| 行尾 | 每行末尾有多余 trailing `\t`,解析时剔除空尾列 |
| 手续费 | **本文件不含**,导入时按当前账本手续费设置计算 `fee` 并落库;设置页保存手续费设置后自动重算当前账本历史成交手续费 |

### 健壮性要求
1. 解析器先按 **GB18030 + Tab** 读;若检测到 `<table>`/`<html>` 则 fallback 用 `pandas.read_html()`(兼容其他券商的 HTML 伪装 .xls)。
2. 导入字段采用 **adapter 固定适配**:每个券商/导出模板显式声明可接受表头,不使用 AI 或泛化语义猜测;`价格`、`数量`、`金额` 这类模糊表头不自动映射到成交字段。
3. 未识别模板返回检测到的表头与最接近模板缺失字段,作为后续「字段映射预览」入口;在用户确认映射前不入库。
4. **导入去重**:对每条成交以 `(trade_date, stock_code, side, price, quantity, amount)` 做唯一约束;并对整文件计算 `file_hash` 防止整批重复导入。
5. 解析失败的行单独收集报错返回,不影响其余行入库。

## 6. 数据模型设计

> 两层存储:`raw_import_rows` 保留原始行(追溯/重新解析),`trades` 为标准化的单一事实源。持仓与盈亏由 `trades` **动态计算**,持仓可选做快照缓存。

### 6.0 accounts(账本) + 多账本数据隔离

> **账本 ≠ 用户**:`accounts` 是同机同人下的多个独立交易账本(模拟盘/信用户/另一券商户/另一套策略),不做登录鉴权(见 PRD 非目标)。

**accounts(交易账本)**
| 字段 | 类型 | 说明 |
|------|------|------|
| id | PK | |
| name | str(50) | 账本名,如「模拟数据」「账本1」「账本2」 |
| kind | enum(live/demo) | 兼容字段,前端不展示,不参与业务逻辑 |
| is_default | bool | 是否默认账本;无显式选择时落到此账本 |
| sort_order | int | 切换器排序 |
| commission_rate | Numeric(18,8) | 当前账本佣金费率,默认 `0.0004`(万4),买卖双向;上限 `0.003`(千3) |
| commission_min_fee_exempt | bool | 是否免最低 5 元佣金;默认 false,即单笔佣金最低 5 元 |
| created_at | datetime | |

**账本归属(per-account)** — 以下表新增 `account_id`(FK accounts,NOT NULL):
`trades`、`import_batches`、`cash_flows`、`trade_intents`、`review_reports`、`rule_versions`、`tags`。
`raw_import_rows` 通过 `import_batch_id` 间接归属,不直接加列。

**全局共享(无 account_id)** — 行情与机器级配置,各账本复用以省 TuShare 额度:
`market_daily_bars`、`market_sentiment_snapshots`、`trading_calendar_days`、`stock_list`、`market_sync_state`、`app_settings`(含大模型配置)。

**唯一约束随账本调整**:
- `import_batches.file_hash`:由全局 unique 改为 **`(account_id, file_hash)`** 复合 unique——同一 `.xls` 可分别导入不同账本,互不视为重复。
- `trades`:唯一约束改为 **`(account_id, trade_date, stock_code, side, price, quantity, amount)`**;逐行去重在「账本内」生效。
- `tags.name`:由全局 unique 改为 **`(account_id, name)`** unique。

**请求级账本上下文**:前端在请求头携带 `X-Account-Id`(或 query `account_id`),后端 `deps.py` 解析为当前账本;缺省回退到 `is_default` 账本。所有业务查询/写入按 `account_id` 过滤。行情类接口忽略账本上下文。

**默认与演示**:首次启动若 `accounts` 为空,自动建一个 `is_default=True` 的「模拟数据」账本;Alembic 迁移把存量业务数据回填到该默认账本。真实 `*.db` 不入库(见 §13)。

| 字段 | 类型 | 说明 |
|------|------|------|

| side | enum(buy/sell/transfer_in) | 见下方映射 |
| price | Numeric(18,4) | 成交价格 |
| quantity | int | 成交数量 |
| amount | Numeric(18,4) | 成交金额(权威) |
| fee | Numeric(18,4) | 手续费,导入时按账本佣金费率 + A 股固定费率计算 |

| source | str | 数据来源,如 `ths_xls` |
| remark | str null | 摘要 |
| raw_row_id | FK null | 对应 raw_import_rows |
| import_batch_id | FK | |
| created_at | datetime | |

唯一约束:`(account_id, trade_date, trade_time, stock_code, side, price, quantity, amount)`

买卖标志映射:`证券买入→buy`、`证券卖出→sell`、`担保品划入→transfer_in`

> **担保品划入 ≠ 买入**:信用账户的"担保品划入"是持仓划入/期初持仓事件,映射为 `transfer_in`——**计入持仓数量但不计入买入资金投入**,成本基准按划入价记;绝不可当作 `buy`,否则虚增投入本金、算错盈亏。
> **成交时间口径**:`ths_contract_export` 可还原到秒级成交时间;旧模板仍仅日期级。同日同票多笔买卖按 `trade_time → seq → id` 排序,旧文件无时间时只能保留文件行序。

## 7. 盈亏计算逻辑

- **持仓均价(摊薄成本)**:持仓均价 = 该票净投入现金 ÷ 持仓股数。买入累加(成交额+费),担保品划入按划入价累加,卖出扣减卖出净收入(成交额−费),持股归零时净投入重置为 0。已实现盈亏因此折进剩余股成本,与券商「摊薄成本价/保本价」口径一致。
- **已实现盈亏**:底层保留 **FIFO**(先进先出)逐笔配对,卖出时与最早买入批次结算,用于逐笔对账(对账差=0)。绩效汇总中「累计已实现盈亏」仅统计已清仓股票,持仓中股票的已实现已并入持仓盈亏,避免重复。
- **持仓盈亏(原浮动)**:持仓数量 × (最新价 − 持仓均价),含该票内部已实现盈亏;最新价取自行情服务。
- **当日盈亏**:持仓数量 × (最新价 − 昨收价)。
- **担保品划入(`transfer_in`)**:增加持仓数量,成本按划入价计入持仓成本,但**不计入买入资金投入统计**,自身不产生已实现盈亏。
- 所有金额运算使用 `Decimal`,展示层再格式化。

## 8. 全局技术约束(铁律)

1. **金额/价格/盈亏一律 `Decimal`(SQLAlchemy `Numeric`),禁止 `float`**——避免账目对不上。
2. **证券代码一律字符串**,禁止整型(前导零)。
3. **SQLite 开启 WAL**:启动时执行 `PRAGMA journal_mode=WAL;`。
4. **Alembic 从第一天接入**,所有表结构变更走迁移。
5. SQLAlchemy 使用 **2.0 风格**(`Mapped[]` / `mapped_column`)。
6. 外部数据源(TuShare/akshare)统一封装、带重试与 fallback,不在业务层直接调用。

## 9.5 统计接口口径

### 9.5.1 周换手率趋势（`GET /api/stats/turnover`）
- 返回当前账本近 12 个自然周数据，按 ISO 周展示。
- 周换手率 = （本周买入金额 + 卖出金额）/ 本周平均持仓市值。
- 本周平均持仓市值按周内每日持仓市值平均；价格优先读取 `market_daily_bars` 本地日线收盘价，缺失时回退该持仓的成本价，避免外部行情调用。
- 分母为 0 时 `turnover_rate=null`，前端折线断开，不显示误导性 0% 或无限大。
- 返回字段包含：`week/week_start/week_end/volume/avg_holding_value/buy_amount/sell_amount/buy_count/sell_count/trade_count/turnover_rate/level/warning`。
- 阈值等级：`normal` <80%；`frequent` 80%-150%；`high` ≥150%。前端用绿/黄/红分段折线与 80%、150% 虚线展示，偏频繁及以上提示「操作频繁」。
- 复盘统计页上方「收益率走势」不由 `/api/stats/turnover` 计算；前端使用同账本 `/api/positions/equity-curve`，按每个周区间 `week_end` 查找交易总览「收益率走势」同口径的账户收益率点（找不到精确日期时取 `week_end` 前最近点），避免复盘页维护第二套周收益率公式。

## 13. Git 数据治理(本地数据库是否上传)

> 结论:**真实业务数据库不上传 GitHub**。Git 存「代码 + 文档 + 可复现的演示种子」,本机存「运行态数据 + 行情缓存 + 真实导入文件」。

| 数据 | 是否进 Git | 说明 |
|------|-----------|------|
| `backend/tradingreview.db`(`*.db` / `*.db-wal` / `*.db-shm`) | ❌ | 含真实资金/成交隐私,体积会增长,多人协作易冲突;已被 `backend/.gitignore` 忽略 |
| 行情缓存(`market_daily_bars` 等,在 db 内) | ❌ | 随 db 一并忽略;可本地拉取,体积大 |
| 同花顺导出 `.xls`、`tradinghistory/` | ❌ | 个人流水敏感;根 `.gitignore` 已忽略 |
| `.env`(TuShare token / LLM api_key) | ❌ | 已忽略;`.env.example` 给字段模板 |
| 演示数据 | ✅(以脚本形式) | 不提交 `.db`,而提交种子脚本 `app/seed_demo.py`:生成 `kind=demo` 的「模拟账户」+ 虚构成交/出入金/规则;他人 clone 后 `迁移 → seed → (可选)拉行情` 即可看到完整 UI |

**口径**:本机 = 运行态(SQLite + 行情缓存 + 真实导入);Git = 代码与可复现演示种子,不是备份盘。
