# 开发协作规范

> 变更记录：2026-05-31 — 初版，启用 `main` 分支保护与 PR 合并流程

## 分支保护（已配置）

`main` 分支已启用 GitHub Branch Protection，规则如下：

| 规则 | 说明 |
|------|------|
| 必须通过 Pull Request 合并 | 禁止直接 `git push origin main` |
| 必须通过 CI | `Backend Tests`、`Frontend Lint & Build` 两项均绿 |
| 分支需与 main 同步（strict） | PR 基于最新的 `main`，避免「CI 绿但合入后冲突/失败」 |
| 管理员同样遵守 | `enforce_admins: true`，无例外 bypass |
| 审批人数 | **0**（个人项目，自己 merge 即可，无需他人 review） |
| 禁止 force push / 删除 main | 保护历史记录 |

查看与修改：GitHub 仓库 → **Settings → Branches → Branch protection rules**。

## PR 与不 PR 的区别

### 不 PR（直接 push 到 main）

```
本地改代码 → commit → git push origin main → CI 跑
                              ↑
                    代码已经进 main，CI 只是事后通知
```

| 优点 | 缺点 |
|------|------|
| 步骤少，适合个人快速迭代 | CI 失败时，**坏代码已在 main 上** |
| 无需开 PR、等 merge | 无合并前审查入口，难追溯「为什么合入」 |
| | 无法强制「CI 不绿就不合入」 |

### PR 流程（推荐，当前规范）

```
feat/xxx 分支开发 → push → 开 PR → CI 跑 → 全绿 → Merge → main 更新
                                    ↑
                          不绿则 main 不受影响，修完再 push
```

| 优点 | 缺点 |
|------|------|
| **CI 是门禁**：不通过无法 merge 到 main | 多几步（建分支、开 PR、点 merge） |
| main 始终保持可部署/可拉取状态 | 紧急 hotfix 也要走分支（习惯后很快） |
| PR 页面集中看 diff、CI 结果、讨论 | |
| 每个功能有独立提交历史，方便回滚 | |

**一句话**：不 PR 是「先上车后检票」；PR 是「检票通过才上车」。

## 日常开发流程

### 1. 从 main 拉最新代码

```bash
git checkout main
git pull origin main
```

### 2. 创建功能分支

分支命名：`feat/简短描述`、`fix/简短描述`、`docs/简短描述`（与 commit type 对应）。

```bash
git checkout -b feat/market-sync-ui
```

### 3. 开发、提交

```bash
# … 改代码 …
git add .
git commit -m "feat: add market sync progress bar"
```

Commit 格式：`<type>: <subject>`（type: feat / fix / refactor / docs / style / chore）

### 4. 推送分支并开 PR

```bash
git push -u origin feat/market-sync-ui
gh pr create --title "feat: add market sync progress bar" --body "$(cat <<'EOF'
## Summary
- 行情同步卡片增加进度条

## Test plan
- [ ] 设置页点击「拉取最新行情」，进度条正常更新
- [ ] CI 通过
EOF
)"
```

也可在 GitHub 网页上从 push 提示一键 **Compare & pull request**。

### 5. 等 CI 通过，合并 PR

```bash
gh pr merge --merge   # 或网页点 Merge pull request
```

合并后清理本地分支（可选）：

```bash
git checkout main
git pull origin main
git branch -d feat/market-sync-ui
```

## 紧急修复（hotfix）

流程相同，只是分支名用 `fix/`：

```bash
git checkout main && git pull
git checkout -b fix/ci-test-mock
# … 修复 …
git push -u origin fix/ci-test-mock
gh pr create --title "fix: mock TuShare in CI tests"
# CI 绿 → merge
```

## 本地与 CI 的关系

| 操作 | 是否必须 |
|------|----------|
| 改代码前从 main 拉最新 | 推荐 |
| 本地跑 `pytest`（后端改动时） | 推荐，减少 CI 失败轮次 |
| 本地跑 `pnpm build`（前端改动时） | 推荐 |
| 本地模拟 GitHub Actions v6 环境 | **不需要**，以 CI 结果为准 |
