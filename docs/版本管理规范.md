# 版本管理规范

> 变更记录：2026-05-31 — 初版，确立 SemVer + Git Tag + GitHub Release 流程

## 1. Git Tag 是什么

**Git Tag** 是指向某一固定提交的不可变引用，相当于给该提交贴上永久性版本标签。

| 概念 | 说明 |
|------|------|
| Commit | 开发过程中的代码快照，分支指针随新提交向前移动 |
| Tag | 永远指向特定提交，不随分支移动，用于标记发布版本 |

在本项目中，Tag 与 **GitHub Release** 配合使用：Tag 标记代码位置，Release 提供版本说明与（可选）附件。

---

## 2. Tag 类型

| 类型 | 命令示例 | 适用场景 |
|------|----------|----------|
| **轻量标签** | `git tag v0.1.0` | 临时标记、个人本地 |
| **附注标签** | `git tag -a v0.1.0 -m "..."` | **正式发布（本项目采用）** |

附注标签包含创建者、时间、说明信息，便于追溯，**所有对外 Release 必须使用附注标签**。

---

## 3. 语义化版本（SemVer）

格式：**主版本号.次版本号.修订版本号**（`MAJOR.MINOR.PATCH`）

```
v0.1.0
 │ │ └── PATCH：向后兼容的问题修复
 │ └──── MINOR：向后兼容的功能新增
 └────── MAJOR：不向后兼容的变更（破坏性升级）
```

### 3.1 本项目阶段说明

当前处于 **0.y.z** 阶段（早期产品），表示 API 与数据模型仍可能调整。达到稳定对外承诺后再发布 **v1.0.0**。

| 版本变化 | 何时递增 | 示例 |
|----------|----------|------|
| PATCH | Bug 修复、文档修正、无行为变化的 chore | `v0.1.0` → `v0.1.1` |
| MINOR | 新增功能，现有用法仍兼容 | `v0.1.1` → `v0.2.0` |
| MAJOR | 破坏性变更（数据库迁移不兼容、API 删除字段等） | `v0.x.x` → `v1.0.0` |

### 3.2 预发布版本（可选）

复杂功能上线前可使用：

| 后缀 | 含义 |
|------|------|
| `-alpha.N` | 内部测试，功能不完整 |
| `-beta.N` | 公开试用，功能大体完成 |
| `-rc.N` | 发布候选，无重大已知问题即可转正 |

示例：`v0.2.0-beta.1`。预发布版本排在同号正式版之前（`v0.2.0-beta.1` < `v0.2.0`）。

### 3.3 命名约定

- 统一使用小写 **`v` 前缀**：`v0.1.0`（不用 `V1.0.0` 或 `release-2026-05`）
- 不使用空格或特殊符号
- Tag 名与 GitHub Release 标题一致

## 4. Tag 与分支、PR 的关系

```
feat/xxx ──PR──▶ main ──打 Tag──▶ v0.1.0 ──GitHub Release──▶ 对外发布说明
```

| 机制 | 作用阶段 | 关系 |
|------|----------|------|
| **PR** | 开发合并 | 代码进入 `main` 的门禁（CI 必须通过） |
| **Tag** | 发布标记 | 指向 `main` 上某一已验证的提交 |
| **Release** | 对外分发 | 基于 Tag，附带 Release Notes |

Tag 不依赖分支存在：即使功能分支已删除，Tag 仍指向原提交。

## 5. 发布流程（本项目）

### 前置条件

- [ ] 目标功能已在 `main` 上（经 PR 合并）
- [ ] CI 全绿（Backend Tests + Frontend Lint & Build）
- [ ] 本地/手动冒烟测试通过（个人项目推荐）
- [ ] `CHANGELOG.md` 已更新对应版本条目
- [ ] 如需改需求文档，已同步 `docs/` 相关规范

### 步骤

**1. 确认版本号**

根据变更类型按 SemVer 递增（见 §3.1）。

**2. 更新 CHANGELOG**

在 `CHANGELOG.md` 顶部追加新版本区块，归类为 Added / Changed / Fixed 等。

**3. 合并 CHANGELOG 到 main**（若尚未合并）

```bash
git checkout main && git pull
git checkout -b chore/release-v0.x.x
# 编辑 CHANGELOG.md
git commit -m "chore: prepare release v0.x.x"
git push -u origin chore/release-v0.x.x
gh pr create && # CI 绿后 merge
```

**4. 在 main 最新提交上创建附注 Tag**

```bash
git checkout main && git pull
git tag -a v0.x.x -m "v0.x.x: 一句话摘要"
git push origin v0.x.x
```

**5. 创建 GitHub Release**

```bash
gh release create v0.x.x \
  --title "v0.x.x" \
  --notes-file CHANGELOG.excerpt.md
```

或在 GitHub 网页：**Releases → Create a new release → Choose tag**，粘贴 CHANGELOG 对应段落。

### 发布原则

- **先合代码、后打 Tag**：Tag 永远打在已合并且 CI 通过的 `main` 提交上
- **先 Push 代码、再 Push Tag**：顺序不可颠倒
- **Tag 不可随意移动**：打错位置见 §7 修正流程
- 个人项目暂不强制 GPG 签名；若仓库公开且多人协作，后续可启用

## 6. 常用命令速查

```bash
# 列出本地标签
git tag

# 查看标签详情
git show v0.1.0

# 创建附注标签
git tag -a v0.1.0 -m "v0.1.0: 首个可用版本"

# 推送单个标签
git push origin v0.1.0

# 推送所有标签
git push origin --tags

# 删除本地标签
git tag -d v0.1.0

# 删除远程标签
git push origin --delete v0.1.0

# 检出某标签（只读查看，detached HEAD）
git checkout v0.1.0
```

## 7. 常见问题

### Tag 打错提交

```bash
git tag -d v0.1.0
git push origin --delete v0.1.0
git tag -a v0.1.0 -m "..." <正确提交的 SHA>
git push origin v0.1.0
```

若 GitHub Release 已创建，需同步删除后重建。

### 推送 Tag 提示已存在

远程已有同名 Tag 时，先删除远程旧 Tag（确认无 Release 依赖），再重新推送。

### 与 CI 的关系

- **PR 阶段**：CI 验证待合并代码
- **Tag / Release 阶段**：当前 CI 不在打 Tag 时自动触发额外部署；未来若接入打包发布，可在 `.github/workflows/` 增加 `on: push: tags: ['v*']` 工作流
